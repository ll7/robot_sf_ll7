# Issue #3484 — Feasibility Diagnostics

This note records the diagnostic-only runner that probes whether universally failing classic
scenario families look route-infeasible, vehicle-infeasible, or still unresolved after simple
scripted rollouts.

## Scope

`robot_sf/scenario_certification/feasibility_diagnostics.py` implements a small report builder for
`bottleneck`, `cross_trap`, and `head_on_corridor` scenarios. The report schema is
`scenario_feasibility_diagnostics.v1` and its claim boundary is
`diagnostic_only_not_benchmark_evidence`.

The runner executes:

- static route-clearance certification through `certify_scenario`;
- actor-free rollout by removing pedestrian actors in memory;
- scripted/oracle trajectory rollout through the configured deterministic `oracle_algo`.

The extended-time lane is represented in the report and remains `null` by default for this slice.
Call the CLI with `--include-extended-time` when that lane is intentionally part of a local smoke
probe.

## Command

```bash
uv run python scripts/tools/run_feasibility_diagnostics_issue_3484.py \
  --scenario-config configs/scenarios/classic_interactions.yaml \
  --family bottleneck \
  --family cross_trap \
  --family head_on_corridor \
  --output docs/context/evidence/issue_3484_feasibility_diagnostics_2026-07-02/diagnostic_report.json
```

The output includes per-scenario lane rows and a per-family verdict table generated by the existing
`scenario_failure_cause.v1` classifier. Verdicts remain fail-closed when a lane is blocked or not
run.

## Boundaries

This is not a full benchmark campaign, not a planner ranking update, and not a paper or dissertation
claim edit. The report is meant to identify the next empirical action for scenario validity and
planner-limitation triage.
